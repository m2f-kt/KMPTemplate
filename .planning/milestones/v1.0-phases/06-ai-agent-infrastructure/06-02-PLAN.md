  <action>
    Create an annotation-based ToolSet that gives agents access to user data from the database. This follows the Koog annotation-based tool pattern from research.

    ```kotlin
    package com.m2f.server.ai.tools

    import ai.koog.agents.core.tools.annotations.LLMDescription
    import ai.koog.agents.core.tools.annotations.Tool
    import ai.koog.agents.core.tools.ToolSet
    import com.m2f.server.auth.repository.UserRepository

    @LLMDescription("Tools for querying user information from the application database")
    class UserTools(
        private val userRepository: UserRepository,
    ) : ToolSet {

        @Tool
        @LLMDescription("Look up a user's profile by their email address. Returns user name, email, and role.")
        suspend fun getUserByEmail(
            @LLMDescription("The email address of the user to look up")
            email: String
        ): String {
            val user = userRepository.findByEmail(email)
                ?: return "No user found with email: $email"
            return "User: ${user.name} (${user.email}), role: ${user.role}"
        }

        @Tool
        @LLMDescription("Count the total number of registered users in the system")
        suspend fun getUserCount(): String {
            // UserRepository doesn't have a count method, so this is a simple demo tool
            return "User count functionality available via database queries"
        }
    }
    ```

    IMPORTANT: Check the actual Koog 0.6.2 import paths at compile time. The annotation package may be:
    - `ai.koog.agents.core.tools.annotations.Tool` and `ai.koog.agents.core.tools.annotations.LLMDescription`
    - OR `ai.koog.agents.core.tools.Tool` and `ai.koog.agents.core.tools.LLMDescription`

    Inspect the Koog JAR's package structure if imports fail. The ToolSet interface may be at `ai.koog.agents.core.tools.ToolSet` or similar. Use IDE auto-import or `jar tf` on the resolved artifact to find correct packages.

    The tools accept UserRepository via constructor (Koin-injected in plan 06-03). Tool methods return String because Koog serializes tool results as text for the LLM.
  </action>

    1. Create `ConversationsTable.kt` -- an Exposed table for storing agent checkpoint data:
       ```kotlin
       package com.m2f.server.ai.persistence

       import org.jetbrains.exposed.v1.core.Table
       import org.jetbrains.exposed.v1.core.kotlin.datetime.timestamp

       /**
        * Stores agent conversation checkpoints for persistence across requests.
        * Each row represents a checkpoint for a specific agent (keyed by composite agentId).
        */
       object ConversationsTable : Table("ai_conversations") {
           val id = varchar("id", 255)
           val agentId = varchar("agent_id", 255)
           val checkpointData = text("checkpoint_data") // Serialized JSON
           val createdAt = timestamp("created_at")
           val updatedAt = timestamp("updated_at")

           override val primaryKey = PrimaryKey(id)
       }
       ```
       Use `text` for checkpoint_data (not JSONB) to avoid R2DBC JSONB driver issues per research recommendation.

    2. Create `ExposedPersistenceStorage.kt` -- a custom PersistenceStorageProvider using the project's R2DBC Exposed:

       IMPORTANT: The exact PersistenceStorageProvider interface must be verified from Koog 0.6.2 source. Research shows:
       ```kotlin
       interface PersistenceStorageProvider {
           suspend fun saveCheckpoint(agentId: String, checkpoint: AgentCheckpointData): String
           suspend fun loadCheckpoint(agentId: String, checkpointId: String): AgentCheckpointData?
           suspend fun getLatestCheckpoint(agentId: String): AgentCheckpointData?
           suspend fun listCheckpoints(agentId: String): List<CheckpointMetadata>
           suspend fun deleteCheckpoint(agentId: String, checkpointId: String)
       }
       ```

       Implement using `suspendTransaction(db = db)` blocks following the UserRepository pattern. Serialize `AgentCheckpointData` to JSON string for storage in the `checkpoint_data` column. Use `kotlinx.serialization.json.Json` for serialization.

       If the `AgentCheckpointData` class does not support kotlinx serialization, use `toString()` for saving and explore parsing options -- this is a known open question from research. If serialization is problematic, fall back to storing the raw string representation and document the limitation.

       Generate checkpoint IDs using `kotlin.uuid.Uuid.random().toString()`.

       Constructor takes `R2dbcDatabase` (Koin-injected in plan 06-03).
