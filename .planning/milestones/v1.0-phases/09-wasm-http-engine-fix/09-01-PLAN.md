<objective>
Swap the CIO HTTP engine for the Js engine on the wasmJs target and install CORS on the server, so browser-based WASM builds can make network requests.

Purpose: CIO engine on wasmJs requires Node.js `net` module which is unavailable in browser environments. The Js engine uses the browser's native `fetch` API. However, `fetch` enforces CORS policy, and the server currently has no CORS configuration -- so both changes are required together.

Output: Working HTTP networking from WASM browser target to Ktor server, with CORS properly configured for development.
</objective>

    import io.ktor.client.engine.HttpClientEngineFactory

    This changes the import from `io.ktor.client.engine.cio.CIO` to `io.ktor.client.engine.js.Js` and returns `Js` instead of `CIO`.

The Js engine uses the browser's native `fetch` API which works in all browser environments. The `Js` factory is `HttpClientEngineFactory<JsClientEngineConfig>` -- same interface as CIO, OkHttp, and Darwin -- so it drops into the existing expect/actual pattern with no changes to common code.

<task type="auto">
  <name>Task 2: Install CORS plugin on server for browser fetch compatibility</name>
  <files>
    server/src/main/kotlin/com/m2f/template/Application.kt
  </files>
  <action>
    **Why this is needed:** The Js engine uses the browser `fetch` API, which enforces CORS policy. CIO previously bypassed this because it used Node.js raw sockets. Without CORS headers, every browser request will fail with `Access-Control-Allow-Origin` errors even though the engine swap itself is correct. The `ktor-server-cors` dependency is already on the server classpath via the `ktor-security` bundle in `server/build.gradle.kts`.

    **Install the CORS plugin in `Application.module()`.**
    In `server/src/main/kotlin/com/m2f/template/Application.kt`, add the CORS import and install the plugin.

    Add these imports at the top of the file:
    ```kotlin
    import io.ktor.server.plugins.cors.routing.CORS
    import io.ktor.http.HttpHeaders
    ```
    Note: `HttpMethod` is already imported.

    Add `install(CORS) { ... }` in `Application.module()` **before** `configureSecurity()` (after the existing plugin installs -- Resources, ContentNegotiation, WebSockets). Place it as follows:

    ```kotlin
    install(CORS) {
        // Development origins (WASM dev server + backend)
        allowHost("localhost:8080", schemes = listOf("http"))
        allowHost("localhost:8081", schemes = listOf("http"))
        allowHost("localhost:3000", schemes = listOf("http"))
        // HTTP methods used by API endpoints
        allowMethod(HttpMethod.Get)
        allowMethod(HttpMethod.Post)
        allowMethod(HttpMethod.Put)
        allowMethod(HttpMethod.Delete)
        allowMethod(HttpMethod.Options)
        // Headers used by API (Content-Type for JSON, Authorization for Bearer tokens)
        allowHeader(HttpHeaders.ContentType)
        allowHeader(HttpHeaders.Authorization)
        // Allow credentials (needed for auth flows)
        allowCredentials = true
    }
    ```

    **Configuration notes:**
    - Multiple localhost ports are allowed for development flexibility (8080 for backend direct, 8081 and 3000 for common WASM/webpack dev server ports)
    - `allowCredentials = true` is needed because the auth flow uses `Authorization` header with bearer tokens
    - `allowMethod(HttpMethod.Options)` ensures CORS preflight requests succeed
    - Do NOT use `anyHost()` -- this is a template that should model secure defaults
    - Production domains can be added later via environment configuration
  </action>
