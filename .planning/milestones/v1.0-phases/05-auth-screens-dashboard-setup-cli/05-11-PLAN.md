<objective>
Wire the "remember me" feature end-to-end so login sessions persist across app restart when the checkbox is checked, and expire on restart when unchecked.

Purpose: Three gaps prevent remember-me from working: (1) AppNavHost hardcodes `startDestination = LoginRoute` with no token check on startup, (2) `LoginViewModel.login()` ignores the `rememberMe` flag entirely, (3) `TokenStorage` always persists to disk with no session-only mode. This plan fixes all three gaps in dependency order.

Output: Full remember-me chain works -- tokens checked on startup, rememberMe forwarded through login, session-only mode clears tokens on app close.
</objective>

  <action>
    **TokenStorage.kt -- Add session-only mode:**

    Add a private `var sessionOnly: Boolean = false` field to TokenStorage. This tracks whether the current session's tokens should be cleared on next app start.

    Add a new Settings key `KEY_SESSION_ONLY = "auth_session_only"` in the companion object.

    Modify `saveTokens(accessToken: String, refreshToken: String)` to also accept a `rememberMe: Boolean = true` parameter:
    ```kotlin
    fun saveTokens(accessToken: String, refreshToken: String, rememberMe: Boolean = true) {
        settings[KEY_ACCESS_TOKEN] = accessToken
        settings[KEY_REFRESH_TOKEN] = refreshToken
        settings[KEY_SESSION_ONLY] = !rememberMe
        sessionOnly = !rememberMe
    }
    ```
    The `rememberMe = true` default preserves backward compatibility for `register()` and `refresh()` callers in AuthApi that always persist.

    Modify `clearTokens()` to also clear the session-only flag:
    ```kotlin
    fun clearTokens() {
        settings.remove(KEY_ACCESS_TOKEN)
        settings.remove(KEY_REFRESH_TOKEN)
        settings.remove(KEY_SESSION_ONLY)
        sessionOnly = false
    }
    ```

    Add a new function `isSessionOnly(): Boolean`:
    ```kotlin
    fun isSessionOnly(): Boolean = settings.getBoolean(KEY_SESSION_ONLY, false)
    ```

    Add a new function `clearSessionTokens()` that clears tokens ONLY IF the session-only flag is set. This will be called on startup:
    ```kotlin
    fun clearSessionTokens() {
        if (isSessionOnly()) {
            clearTokens()
        }
    }
    ```

    **AuthApi.kt -- Forward rememberMe to TokenStorage:**

    Change `login()` to accept `rememberMe: Boolean` parameter:
    ```kotlin
    suspend fun login(request: LoginRequest, rememberMe: Boolean = true): Either<AppError, AuthResponse> =
        apiCall<AuthResponse> {
            client.post("/api/auth/login") {
                contentType(ContentType.Application.Json)
                setBody(request)
            }
        }.onRight { response ->
            tokenStorage.saveTokens(response.accessToken, response.refreshToken, rememberMe)
        }
    ```

    The `register()` and `refresh()` functions stay unchanged -- they always persist (rememberMe defaults to true).

    **LoginViewModel.kt -- Forward rememberMe from state to AuthApi:**

    Change the `login()` function (line 53) to pass `current.rememberMe`:
    ```kotlin
    authApi.login(LoginRequest(current.email.trim(), current.password), rememberMe = current.rememberMe)
    ```

    This is the only change in LoginViewModel -- the `rememberMe` state field already exists in `LoginState` and the `onRememberMeChange` handler already works.

    **Add startup token check and session cleanup:**

    1. At the top of `AppNavHost()`, inject TokenStorage via koinInject:
    ```kotlin
    val tokenStorage = koinInject<TokenStorage>()
    ```
    Note: `koinInject` import is already present (used in OAuthCallbackRoute). `TokenStorage` import is already present.

    2. Add a `LaunchedEffect(Unit)` block BEFORE the existing OAuth callback LaunchedEffect. This block does two things:
       - Clears session-only tokens (tokens from a login where rememberMe was unchecked)
       - Checks for surviving tokens and navigates to dashboard if found

    ```kotlin
    // Check for existing auth tokens on startup
    LaunchedEffect(Unit) {
        // Clear tokens from sessions where rememberMe was false
        tokenStorage.clearSessionTokens()

        // If tokens survived (rememberMe was true), skip login
        val accessToken = tokenStorage.getAccessToken()
        if (accessToken != null) {
            navController.navigate(DashboardRoute) {
                popUpTo(LoginRoute) { inclusive = true }
            }
        }
    }
    ```

    Place this LaunchedEffect BEFORE the existing OAuth callback LaunchedEffect (which starts at the current line 36). The ordering matters: session cleanup and token check should happen first, then OAuth callback check. If both an OAuth callback AND stored tokens exist, the OAuth callback LaunchedEffect will override since it also uses `popUpTo(0) { inclusive = true }`.

    3. The `startDestination = LoginRoute` stays as-is (line 52). LoginRoute is the correct default -- the LaunchedEffect navigates away from it immediately if tokens exist. Using a conditional startDestination would cause recomposition issues.

    **Do NOT change anything else in AppNavHost.** The OAuth callback handling, LoginRoute, RegisterRoute, DashboardRoute, ProfileRoute, and ForgotPasswordRoute composable blocks remain unchanged.
