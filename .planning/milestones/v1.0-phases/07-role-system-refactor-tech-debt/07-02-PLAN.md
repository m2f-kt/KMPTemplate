1. Create `RolesTable.kt` in `server/auth/src/main/kotlin/com/m2f/server/auth/tables/`:
```kotlin
object RolesTable : Table("roles") {
    val id = integer("id").autoIncrement()
    val name = varchar("name", 50).uniqueIndex()
    val level = integer("level")
    override val primaryKey = PrimaryKey(id)
}
```
Follow existing table conventions (same package, same import style as UsersTable).

2. Update `UsersTable.kt`:
- Replace `val role = varchar("role", ROLE_LENGTH).default("USER")` with `val roleId = integer("role_id").references(RolesTable.id).default(1)` (1 = USER role ID from seeded data).
- Remove the `ROLE_LENGTH` constant (no longer needed).
- Add import for `RolesTable`.

3. Add a new migration in `Auth.kt` for the roles table and users FK migration. Create class `CreateRolesTableAndMigrateUsersMigration` implementing `Migration`:
- version: `"20260215000001"`
- description: `"Create roles table, seed roles, migrate users.role to role_id FK"`
- migrate() steps:
  a. `SchemaUtils.create(RolesTable)` -- create the roles table
  b. Seed 3 rows: `RolesTable.insert { it[name] = "USER"; it[level] = 0 }`, same for ADMIN (level 1), POWER_ADMIN (level 2)
  c. For the FK migration on existing users data, use a pragmatic approach: since this is a template project (not a production migration with millions of rows), the migration can use `SchemaUtils.createMissingTablesAndColumns(UsersTable)` after updating UsersTable's column definition. This will add the `role_id` column with default 1 (USER).

IMPORTANT NOTE on migration strategy: The research flagged that `addMissingColumnsStatements()` may be unreliable with R2DBC. Since this is a template (users table is typically empty or has test data), the simplest reliable approach is:
- The new `UsersTable` definition has `roleId` with `.default(1)` -- any existing rows will get `role_id = 1` (USER) when the column is added.
- Use `SchemaUtils.createMissingTablesAndColumns(UsersTable)` which handles both creating missing columns and adding constraints.
- The old `role` varchar column will remain in the DB but be ignored by Exposed (Exposed only maps declared columns). This avoids risky ALTER TABLE DROP COLUMN over R2DBC. A future cleanup migration can drop it if needed.

This follows Approach B from the research: keep the varchar physically in the DB (harmless), but all application code uses `roleId` FK going forward.

4. Register the new migration in `registerAuthMigrations()` -- add it after the existing migrations:
```kotlin
MigrationRegistry.register(CreateRolesTableAndMigrateUsersMigration())
```

Refactor all server-side code to use `UserRole` from `com.m2f.template.models.UserRole` instead of String roles. Changes file by file:

**UserRepository.kt:**
- Change `UserRecord.role` from `String` to `UserRole`.
- Change `insert()` parameter `role: String` to `role: UserRole`.
- In `insert()`, write `it[UsersTable.roleId] = roleIdForRole(role)` instead of `it[UsersTable.role] = role`. The helper function `roleIdForRole` maps UserRole variants to their seeded ID: User->1, Admin->2, PowerAdmin->3. Implement this as a private function or a `when` expression inline.
- In `toUserRecord()`, read `this[UsersTable.roleId]` and map back to `UserRole` using a lookup. Since the IDs are known (1=USER, 2=ADMIN, 3=POWER_ADMIN), use a `when` expression: `role = when(this[UsersTable.roleId]) { 2 -> UserRole.Admin; 3 -> UserRole.PowerAdmin; else -> UserRole.User }`.
- Alternatively (and preferably), to avoid hardcoded ID mapping, join with RolesTable and read the name: `role = UserRole.fromString(this[RolesTable.name])`. BUT this requires a join in every query, adding complexity. For a template project, the hardcoded ID mapping is acceptable and simpler. Use the `when` on roleId approach.
