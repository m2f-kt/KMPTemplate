3. **AuthService.kt** — Modify to accept invitation after registration:
   - Add `invitationService: InvitationService` constructor parameter
   - After user creation (Step 5), if `request.invitationToken != null`:
     a. Validate invitation exists and is not expired/accepted (call getInvitation first)
     b. Call `invitationService.acceptInvitation(AcceptInvitationRequest(request.invitationToken), userId.toString())`
     c. If invitation fails, the entire registration should fail (transactional behavior via Raise)
   - Import required types: `AcceptInvitationRequest`, `InvitationService`

**Critical:** The invitation validation MUST happen BEFORE user creation to avoid orphan users. Restructure to:
- Step 1: Validate all fields (existing)
- Step 2: If invitationToken present, validate invitation (not expired, not accepted)
- Step 3: Check for duplicate email (existing)
- Step 4-8: Create user and tokens (existing)
- Step 9 (new): If invitationToken present, call acceptInvitation to link user to group

The validation in Step 2 should use a new helper that just validates without accepting:
```kotlin
private suspend fun validateInvitation(token: String): InvitationRecord = raise.withError(...) {
    val invitation = invitationRepository.findByToken(token) ?: raise(InvitationNotFound())
    ensure(invitation.acceptedAt == null) { InvitationAlreadyAccepted() }
    val nowLocal = Clock.System.now().toLocalDateTime(TimeZone.UTC)
    ensure(nowLocal < invitation.expiresAt) { InvitationExpired() }
    invitation
}
```

Actually, simpler approach: Just call acceptInvitation after user creation. If it fails, the whole request fails (Raise propagates error). User sees clear error. No orphan users because:
- Email validation happens first
- If invitation is invalid, error is thrown after user creation BUT before response
- The database transaction should roll back (verify Exposed R2DBC behavior)

If rollback doesn't work automatically, consider: validate invitation BEFORE creating user.
