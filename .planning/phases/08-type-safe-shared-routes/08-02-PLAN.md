<objective>
Migrate all server-side HTTP routes from string-based `route("/path") { get("/sub") }` to type-safe `get<Resource>` / `post<Resource>` handlers using the @Resource classes defined in Plan 01.

Purpose: Eliminate all hardcoded route strings from server HTTP handlers, providing compile-time safety. If a route path changes in ApiRoutes.kt, the compiler will flag every server handler that needs updating.

Output: All 13 server HTTP API routes use type-safe handlers. String-based routing remains only for OAuth (browser redirects, not API), WebSocket (KTOR-4369, but uses constant), and /amazing (demo endpoint).
</objective>

  <files>

  <action>
    **AuthRoutes.kt migration:**

    Replace the entire `route("/api/auth") { ... }` block with flat type-safe handlers. The `route()` wrapper is no longer needed because each `@Resource` class encodes the full path.

    IMPORTANT: Use `import io.ktor.server.resources.post` (NOT `io.ktor.server.routing.post`). The server-resources variant of `post` accepts a type parameter `post<R> { ... }`.

    Before:
    ```kotlin
    route("/api/auth") {
        post("/register") { conduit(HttpStatusCode.Created) { ... } }
        post("/login") { conduit { ... } }
        ...
    }
    ```

    After:
    ```kotlin
    post<Auth.Register> { conduit(HttpStatusCode.Created) { ... } }
    post<Auth.Login> { conduit { ... } }
    post<Auth.Refresh> { conduit { ... } }
    post<Auth.ForgotPassword> { conduit { ... } }
    post<Auth.ResetPassword> { conduit { ... } }
    authenticate {
        post<Auth.Logout> { conduitAuth { userId -> ... } }
    }
    ```

    Import changes:
    - Add: `import io.ktor.server.resources.post` (type-safe server resources post)
    - Add: `import com.m2f.template.models.routes.Auth`
    - Remove: `import io.ktor.server.routing.post` and `import io.ktor.server.routing.route`

    The `conduit` and `conduitAuth` helper calls inside the lambda bodies remain UNCHANGED. They use the same `RoutingContext` receiver that both string-based and resource-based handlers provide.

    The `authenticate { }` block for logout stays the same -- `authenticate` is a routing scope decorator that works with both string and resource routes.

    **UserRoutes.kt migration:**

    Replace `route("/api/users") { ... }` with flat type-safe handlers.

    Before:
    ```kotlin
    authenticate {
        route("/api/users") {
            get("/me") { conduitAuth { ... } }
            put("/me") { conduitAuth { ... } }
            withRole(UserRole.Admin) {
                get("/{id}") { conduitAuth { _ -> val targetId = getStringParam("id"); ... } }
            }
        }
    }
    ```

    After:
    ```kotlin
    authenticate {
        get<Users.Me> { conduitAuth { userId -> userService.getProfile(userId) } }
        put<Users.Me> { conduitAuth { userId -> val request = getModel<UpdateProfileRequest>(); userService.updateProfile(userId, request) } }
        withRole(UserRole.Admin) {
            get<Users.ById> { resource ->
                conduitAuth { _ ->
                    userService.getUserById(resource.id)
                }
            }
        }
    }
    ```

    Key changes for UserRoutes:
    - `get<Users.ById> { resource -> ... }` receives the deserialized resource as lambda parameter. Use `resource.id` instead of `getStringParam("id")`.
    - Remove `import io.ktor.server.routing.get`, `import io.ktor.server.routing.put`, `import io.ktor.server.routing.route`
    - Add `import io.ktor.server.resources.get`, `import io.ktor.server.resources.put`
    - Add `import com.m2f.template.models.routes.Users`
    - Remove `import com.m2f.core.config.server.getStringParam` (no longer needed -- path param comes from resource)
    - The `withRole(UserRole.Admin)` block wrapping `get<Users.ById>` should continue to work -- it installs a transparent route selector that does not interfere with resource routing.
  </action>
  <verify>Run `./gradlew :server:auth:compileKotlin` -- should compile without errors, confirming conduit/conduitAuth work inside type-safe handlers and withRole is compatible.</verify>
