**1. Migrate ValidationSupport.kt:**

Change the FieldError messages in validation functions from hardcoded English strings to StringKey code values that can be resolved at the UI layer:

```kotlin
// Before: FieldError("email", "Email must not be blank")
// After:  FieldError("email", StringKey.VALIDATION_EMAIL_BLANK.code)
```

The `FieldError.message` field stays as `String` (wire compatibility) but now contains the StringKey code instead of an English message. ViewModels will map this to StringKey.

Update all four validation functions: validateEmail, validatePassword, validateName, validateRequired.

**2. Migrate LoginModel + LoginViewModel + LoginMutation:**

- LoginModel: Change `emailError: String? = null` to `emailError: StringKey? = null`, same for `passwordError` and `serverError`
- LoginMutation: Change `SetValidationErrors(emailError: String?, passwordError: String?)` to `SetValidationErrors(emailError: StringKey?, passwordError: StringKey?)`
- LoginMutation: Change `SetServerError(error: String)` to `SetServerError(error: StringKey)`
- LoginViewModel.handleLogin(): Replace inline validation strings with StringKey values:
  ```kotlin
  val emailError = when {
      current.email.isBlank() -> StringKey.VALIDATION_EMAIL_BLANK
      !current.email.contains("@") || !current.email.contains(".") -> StringKey.VALIDATION_EMAIL_INVALID
      else -> null
  }
  ```
- LoginViewModel server error handling: Map `error.code` to StringKey:
  ```kotlin
  ifLeft = { error ->
      val key = StringKey.fromCode(error.code) ?: StringKey.GENERIC_ERROR
      sendMutation(LoginMutation.SetServerError(key))
  }
  ```

**3. Migrate RegisterModel + RegisterViewModel + RegisterMutation:**

- RegisterModel: Change `fieldErrors: Map<String, String>` to `fieldErrors: Map<String, StringKey>`
- RegisterModel: Change `serverError: String?` to `serverError: StringKey?`
- RegisterMutation: Update SetFieldErrors and SetServerError mutation types accordingly
- RegisterViewModel: Map validation FieldError.message to StringKey via `StringKey.fromCode(it.message) ?: StringKey.GENERIC_ERROR`
- RegisterViewModel: Map server error.code to StringKey

**4. Migrate ForgotPasswordModel + ForgotPasswordViewModel + ForgotPasswordMutation:**

- ForgotPasswordModel: Change `emailError: String?` to `emailError: StringKey?`, `serverError: String?` to `serverError: StringKey?`
- ForgotPasswordMutation: Update types accordingly
- ForgotPasswordViewModel: Map validation and server errors to StringKey

**Pattern for all three:** Error fields in Model become `StringKey?`, ViewModels produce StringKey values, Mutations carry StringKey types. Server errors are mapped via `StringKey.fromCode(error.code)`.
