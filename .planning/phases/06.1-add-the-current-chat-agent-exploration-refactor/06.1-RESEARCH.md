# Phase 06.1: Chat Agent Streaming Refactor - Research

**Researched:** 2026-02-14
**Domain:** Koog Custom Strategy Graphs, Streaming API, Input Moderation, Ktor SSE
**Confidence:** MEDIUM -- Custom strategy graph patterns verified from Koog source and reference implementation; moderation limitations confirmed from Koog source; Ktor SSE API verified from official docs. Koog is rapidly evolving (0.6.2 released 2026-02-10), so some patterns may shift.

## Summary

The current `ChatAgentService` uses `chatAgentStrategy()` which has a **confirmed infinite loop bug**: when the LLM responds with a plain assistant message instead of tool calls, the strategy routes to a `giveFeedbackToCallTools` node that instructs the LLM to use tools, and if the LLM still responds with an assistant message, it loops back to itself indefinitely (`giveFeedbackToCallTools forwardTo giveFeedbackToCallTools onAssistantMessage { true }`). This is visible in the source at line 42 of `AIAgentStrategies.kt` in Koog's repository.

The fix is to replace `chatAgentStrategy()` with a custom `strategy()` graph that: (1) accepts plain assistant messages as valid completions, (2) uses `nodeLLMRequestStreaming()` to stream LLM responses as `Flow<StreamFrame>`, and (3) pipes those frames to a Ktor SSE endpoint via a `callbackFlow`/Channel bridge. The reference implementation (`kpavlov/koog-spring-boot-assistant`) demonstrates this exact pattern successfully.

Input moderation via `nodeLLMModerateMessage()` is **not compatible with Google models** -- the `GoogleLLMClient.moderate()` method explicitly throws `UnsupportedOperationException("Moderation is not supported by Google API.")`. The alternative is to use Gemini's built-in safety settings (configured as `LLMParams`) which filter harmful content at the API level, or to implement a custom moderation node that uses a Gemini model with a safety-focused prompt.

**Primary recommendation:** Replace `chatAgentStrategy()` with a custom streaming strategy graph using `nodeLLMRequestStreaming()`. Skip `nodeLLMModerateMessage()` and use Gemini's built-in safety settings for content filtering. Use `AIAgent` (per-request) instead of `AIAgentService` (singleton) for the streaming chat agent. Expose SSE via Ktor's `ktor-server-sse` plugin with `install(SSE)` and `sse()` route handler.

### Pattern 1: Custom Streaming Strategy Graph
**What:** Replace `chatAgentStrategy()` with a custom `strategy()` that uses `nodeLLMRequestStreaming()` and routes assistant messages to finish (instead of looping).
**When to use:** Whenever you need streaming LLM responses and/or the built-in `chatAgentStrategy()` exhibits the infinite loop bug.
**Example:**
```kotlin
// Source: kpavlov/koog-spring-boot-assistant AgentConfiguration.kt + Koog AIAgentNodes.kt source
import ai.koog.agents.core.agent.entity.AIAgentGraphStrategy
import ai.koog.agents.core.dsl.builder.forwardTo
import ai.koog.agents.core.dsl.builder.strategy
import ai.koog.agents.core.dsl.extension.nodeExecuteMultipleTools
import ai.koog.agents.core.dsl.extension.nodeLLMRequestStreaming
import ai.koog.agents.core.dsl.extension.nodeLLMSendMultipleToolResults
import ai.koog.agents.core.dsl.extension.onMultipleToolCalls
import ai.koog.prompt.message.Message
import ai.koog.prompt.message.ResponseMetaInfo
import ai.koog.prompt.streaming.StreamFrame
import kotlinx.coroutines.flow.Flow
import kotlinx.coroutines.flow.map
import kotlinx.coroutines.flow.mapNotNull
import kotlinx.coroutines.flow.toList

fun chatStreamingStrategy(
    processFrame: (StreamFrame.Append) -> Unit
): AIAgentGraphStrategy<String, Any> =
    strategy(name = "chat-streaming") {
        val nodeStreaming by nodeLLMRequestStreaming()
        val executeMultipleTools by nodeExecuteMultipleTools(parallelTools = true)
        val sendToolResults by nodeLLMSendMultipleToolResults()

        // Process streaming frames: emit text chunks, collect tool calls
        val processFrames by node<Flow<StreamFrame>, List<Message.Response>> { frames ->
            var end: StreamFrame.End? = null
            frames
                .mapNotNull { frame ->
                    when (frame) {
                        is StreamFrame.Append -> {
                            processFrame(frame)  // Send to SSE
                            null
                        }
                        is StreamFrame.End -> {
                            end = frame
                            null
                        }
                        is StreamFrame.ToolCall -> frame
                    }
                }
                .map { toolCall ->
                    Message.Tool.Call(
                        id = toolCall.id,
                        tool = toolCall.name,
                        content = toolCall.content,
                        metaInfo = end?.metaInfo ?: ResponseMetaInfo.Empty,
                    )
                }
                .toList()
        }

        // Start -> Stream LLM
        edge(nodeStart forwardTo nodeStreaming)

        // Stream -> Process frames
        edge(nodeStreaming forwardTo processFrames)

        // Process frames -> execute tools (if tool calls present)
        edge(processFrames forwardTo executeMultipleTools onMultipleToolCalls { true })

        // Execute tools -> send results back to LLM
        edge(executeMultipleTools forwardTo sendToolResults)

        // Process frames -> finish (if NO tool calls, i.e., pure text response)
        edge(
            processFrames forwardTo nodeFinish onCondition {
                it.filterIsInstance<Message.Tool.Call>().isEmpty()
            }
        )

        // After sending tool results, loop back for more (optional: could also finish)
        // This handles multi-turn tool usage
    }
```

### Pattern 2: Per-Request AIAgent with callbackFlow
**What:** Create an `AIAgent` instance per request inside a `callbackFlow`, inject the streaming callback into the strategy, and emit text chunks.
**When to use:** When streaming responses per-request where each request needs its own agent lifecycle (creation, run, close).
**Example:**
```kotlin
// Source: kpavlov/koog-spring-boot-assistant ElvenAgent.kt
import ai.koog.agents.core.agent.AIAgent
import ai.koog.agents.core.agent.config.AIAgentConfig
import ai.koog.agents.core.agent.context.RollbackStrategy
import ai.koog.agents.snapshot.feature.Persistence
import kotlinx.coroutines.channels.awaitClose
import kotlinx.coroutines.flow.Flow
import kotlinx.coroutines.flow.callbackFlow
import kotlinx.coroutines.runBlocking

fun streamChat(
    input: String,
    sessionId: String,
    executor: SingleLLMPromptExecutor,
    persistenceStorage: ExposedPersistenceStorage,
    agentConfig: AIAgentConfig,
): Flow<String> = callbackFlow {
    var agent: AIAgent<String, Any>? = null
    var flowClosed = false

    val strategy = chatStreamingStrategy { frame ->
        trySend(frame.text)
    }

    try {
        agent = AIAgent(
            id = sessionId,
            promptExecutor = executor,
            agentConfig = agentConfig,
            strategy = strategy,
        ) {
            install(Persistence) {
                storage = persistenceStorage
                enableAutomaticPersistence = true
                rollbackStrategy = RollbackStrategy.MessageHistoryOnly
            }
        }

        agent.run(input)

        if (!flowClosed) {
            flowClosed = true
            close()
        }
    } catch (e: Exception) {
        if (!flowClosed) {
            flowClosed = true
            trySend("[ERROR] ${e.message}")
            close()
        }
    }

    awaitClose {
        runBlocking {
            try { agent?.close() } catch (_: Exception) {}
        }
    }
}
```

### Pattern 3: Ktor SSE Route
**What:** Use Ktor's SSE plugin to expose a streaming endpoint that collects from a `Flow<String>` and sends `ServerSentEvent`s.
**When to use:** Exposing streaming LLM responses to clients.
**Example:**
```kotlin
// Source: Ktor official docs - https://ktor.io/docs/server-server-sent-events.html
import io.ktor.server.sse.*
import io.ktor.sse.ServerSentEvent

// In Application.module():
install(SSE)

// In routing:
sse("/api/ai/chat/stream") {
    // Authenticate and get request params
    val flow = chatAgentService.streamChat(userId, conversationId, message)
    flow.collect { chunk ->
        send(ServerSentEvent(data = chunk, event = "message"))
    }
    // Send completion event
    send(ServerSentEvent(data = "", event = "done"))
}
```

### Anti-Patterns to Avoid
- **Using `chatAgentStrategy()` as-is:** It has an infinite loop when the LLM responds with plain text instead of tool calls. The `giveFeedbackToCallTools onAssistantMessage { true }` edge creates a self-loop that never terminates.
- **Singleton `AIAgentService` for streaming:** `AIAgentService.createAgentAndRun()` doesn't support injecting per-request streaming callbacks. Use `AIAgent` directly for streaming use cases.
- **Using `nodeLLMModerateMessage()` with Google models:** `GoogleLLMClient.moderate()` throws `UnsupportedOperationException`. This will crash the agent at runtime.
- **Collecting all stream frames then sending:** Defeats the purpose of streaming. Process frames incrementally via `callbackFlow` + `trySend()`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Streaming LLM responses | Custom HTTP chunked encoding | `nodeLLMRequestStreaming()` + Ktor SSE plugin | Koog handles LLM provider-specific streaming; Ktor SSE handles HTTP transport |
| Stream frame processing | Manual parsing of LLM response chunks | `StreamFrame` sealed class (Append/End/ToolCall) | Koog already classifies frames; pattern-match on types |
| Content safety filtering | Custom profanity filter / regex | Gemini built-in safety settings via API parameters | Google's models have built-in harm category detection (harassment, hate speech, sexual, dangerous) |
| Agent lifecycle management | Manual coroutine scope management | `AIAgent` + `callbackFlow` + `awaitClose` | Ensures cleanup on cancellation/error |
| Conversation persistence | Manual serialization/storage | Koog `Persistence` feature + `ExposedPersistenceStorage` | Already implemented in Phase 06; handles checkpoint serialization via `PersistenceUtils.defaultCheckpointJson` |

**Key insight:** The primary custom code needed is the strategy graph definition and the SSE route wiring. Everything else (streaming, persistence, tool execution, frame parsing) is provided by Koog and Ktor.

## Common Pitfalls

### Pitfall 1: chatAgentStrategy() Infinite Loop
**What goes wrong:** When the LLM responds with a plain text message instead of calling a tool (e.g., SayToUser), the strategy enters an infinite loop between `giveFeedbackToCallTools` and itself.
**Why it happens:** The `chatAgentStrategy()` graph has an edge `giveFeedbackToCallTools forwardTo giveFeedbackToCallTools onAssistantMessage { true }` which loops forever when the LLM doesn't cooperate with the "use tools!" instruction.
**How to avoid:** Replace with a custom strategy that accepts assistant messages as valid responses and routes them to `nodeFinish`.
**Warning signs:** Agent hangs indefinitely, `maxAgentIterations` exceeded errors, API costs spike from repeated identical requests.

### Pitfall 2: GoogleLLMClient.moderate() UnsupportedOperationException
**What goes wrong:** Using `nodeLLMModerateMessage()` with a Google model crashes with `UnsupportedOperationException("Moderation is not supported by Google API.")`.
**Why it happens:** `GoogleLLMClient` does not implement the `moderate()` method -- it explicitly throws. Only `OpenAILLMClient` implements moderation via the OpenAI Moderation API.
**How to avoid:** Do not use `nodeLLMModerateMessage()` with Google models. Instead, rely on Gemini's built-in safety settings or implement a custom moderation node that uses a safety-focused Gemini prompt.
**Warning signs:** `UnsupportedOperationException` at runtime when the first message is processed.

### Pitfall 3: SSE Authentication Challenge
**What goes wrong:** SSE endpoints use GET requests with long-lived connections, but the existing auth pattern uses `conduitAuth` inside `post` routes with JWT bearer tokens. SSE clients (like EventSource in browsers) cannot send custom headers.
**Why it happens:** The EventSource browser API does not support custom Authorization headers.
**How to avoid:** For SSE endpoints, accept JWT as a query parameter (`?token=...`) and validate it server-side. Alternatively, use a session cookie approach or require an initial POST to get a short-lived streaming token.
**Warning signs:** 401 errors on SSE connection attempts from browser clients.

### Pitfall 4: Flow Collection Timing with callbackFlow
**What goes wrong:** The `callbackFlow` closes before all frames are sent, or the SSE connection drops before the flow completes.
**Why it happens:** The `agent.run()` call completes, the `callbackFlow` scope ends, but the SSE collection might not have processed all buffered items.
**How to avoid:** Use `awaitClose {}` in the `callbackFlow` to keep it alive until the collector (SSE route) cancels. Do not close the flow in the streaming callback -- let the `onLLMStreamingCompleted` event handler or the agent completion close it.
**Warning signs:** Truncated responses, missing final chunks.

### Pitfall 5: Agent Close / Resource Leak
**What goes wrong:** `AIAgent` instances created per-request are not closed, leaking coroutines and connections.
**Why it happens:** Missing `agent.close()` call when using per-request `AIAgent` creation.
**How to avoid:** Always close the agent in `awaitClose {}` block of the `callbackFlow`, using `runBlocking` if needed (since `awaitClose` has a non-suspend lambda).
**Warning signs:** Memory leaks, growing coroutine count, connection pool exhaustion.

## Open Questions

1. **SSE authentication mechanism**
   - What we know: The existing routes use `authenticate { }` with JWT bearer tokens in Authorization header. SSE (EventSource API in browsers) cannot set custom headers.
   - What's unclear: Whether the project's clients are browser-based (needing EventSource) or native apps (which can set headers on SSE connections).
   - Recommendation: Support both -- accept JWT as query parameter `?token=...` for browser clients, and also accept Authorization header for native clients. The SSE route should validate the token from either source.

2. **Whether to keep the non-streaming POST /api/ai/chat endpoint**
   - What we know: The existing `POST /api/ai/chat` endpoint returns a complete response. Clients may depend on this.
   - What's unclear: Whether to keep both endpoints or replace the POST with SSE-only.
   - Recommendation: Keep both. The POST endpoint is simpler for testing and for clients that don't need streaming. Refactor the POST to use the same custom strategy (non-streaming variant) to also fix the infinite loop bug.

3. **Gemini safety settings granularity**
   - What we know: Gemini supports `HarmCategory` (HARASSMENT, HATE_SPEECH, SEXUALLY_EXPLICIT, DANGEROUS_CONTENT) with blocking thresholds (BLOCK_NONE, BLOCK_LOW_AND_ABOVE, BLOCK_MEDIUM_AND_ABOVE, BLOCK_HIGH_ONLY).
   - What's unclear: How to configure these via Koog's `LLMParams` or `additionalProperties`. The Koog `GoogleLLMClient` may or may not pass safety settings through.
   - Recommendation: Investigate `additionalProperties` in `LLMParams` during implementation. If Koog doesn't pass safety settings through to Google's API, this may require a Koog PR or manual HTTP interception. Classify as LOW confidence.

4. **Strategy output type: `String` vs `Any`**
   - What we know: The reference implementation uses `AIAgentGraphStrategy<String, Any>` with the streaming strategy returning `Any` as output. The existing `chatAgentStrategy()` uses `AIAgentGraphStrategy<String, String>`.
   - What's unclear: Whether the output type matters when the real output is streamed via the callback, not the return value.
   - Recommendation: Use `AIAgentGraphStrategy<String, Any>` to match the reference implementation. The actual response content is delivered via the `processFrame` callback, not the strategy's output.

5. **Tool support in streaming strategy**
   - What we know: The reference implementation includes `nodeExecuteMultipleTools` and `nodeLLMSendMultipleToolResults` for handling tool calls during streaming. The current `ChatAgentService` registers `SayToUser`, `AskUser`, and `ExitTool`.
   - What's unclear: Whether these conversation tools (SayToUser, AskUser) are still needed with the streaming approach, since the LLM's text responses are streamed directly.
   - Recommendation: For the streaming strategy, remove `SayToUser`, `AskUser`, and `ExitTool` tools. The LLM's text output IS the response -- streamed directly to the client. Only add tools that perform real actions (like the `UserTools` from the assistant agent). For the initial refactor, start with no tools in the streaming chat agent.
