    - Add a `commonTest.dependencies` block with `implementation(projects.core.testing)` (currently missing)

    **Read DashboardViewModel.kt and DashboardState.kt** to understand the mock loading pattern (delay(300) + isLoading toggle) and selectNavItem behavior.

    **Create 4 type files** in `app/dashboard/src/commonMain/kotlin/com/m2f/template/app/dashboard/`:

    1. **DashboardIntent.kt** -- sealed interface with:
       - `data object LoadDashboard`
       - `data class NavItemSelected(val item: String)`

    2. **DashboardModel.kt** -- data class replacing DashboardState. Keep the same fields with DashboardMockData defaults:
       ```kotlin
       data class DashboardModel(
           val metrics: List<DashboardMockData.MetricItem> = DashboardMockData.metrics,
           val processes: List<DashboardMockData.ProcessItem> = DashboardMockData.processes,
           val activities: List<DashboardMockData.ActivityItem> = DashboardMockData.activities,
           val deployment: DashboardMockData.DeploymentStatus = DashboardMockData.deployment,
           val isLoading: Boolean = false,
           val userName: String = "user@terminal.dev",
           val selectedNavItem: String = "dashboard",
       )
       ```

    3. **DashboardMutation.kt** -- sealed interface with:
       - `data class SetLoading(val loading: Boolean)`
       - `data class SetNavItem(val item: String)`

    4. **DashboardEvent.kt** -- **empty sealed interface** (no ViewModel-driven navigation):
       ```kotlin
       sealed interface DashboardEvent
       ```
       Logout is handled at Composable level. Profile navigation is a Composable callback.

    **Rewrite DashboardViewModel.kt:**
    - Add constructor parameter `private val sdk: Sdk` (per convention, even though currently unused -- see research open question #2)
    - Extend `MviViewModel<DashboardIntent, DashboardModel, DashboardMutation, DashboardEvent>(initialState = DashboardModel())`
    - `init { take(DashboardIntent.LoadDashboard) }` -- replaces the current init block's direct loading
    - `take(intent)` in `viewModelScope.launch`:
      - `LoadDashboard` -> `sendMutation(SetLoading(true))`, `delay(300)`, `sendMutation(SetLoading(false))` (preserves existing mock loading behavior)
      - `NavItemSelected` -> `sendMutation(SetNavItem(intent.item))`
    - `reduce(model, mutation)` -- pure:
      - SetLoading: `model.copy(isLoading = mutation.loading)`
      - SetNavItem: `model.copy(selectedNavItem = mutation.item)`

    **Update DashboardScreen.kt:** Change `state: DashboardState` to `state: DashboardModel`. Update the import. Verify the internal composables that reference DashboardState also compile -- check DesktopDashboard, MobileDashboard, DashboardSidebar, DashboardBottomNav for state type usage. If they reference the state type by name, update those too.

    **Update AppNavHost.kt DashboardRoute block:**
    - `dashboardViewModel.state` -> `dashboardViewModel.model`
    - `val dashboardState by dashboardViewModel.model.collectAsStateWithLifecycle()` -- keep variable name as `dashboardState` to minimize DashboardScreen call site changes
    - `onNavItemSelected = dashboardViewModel::selectNavItem` -> `onNavItemSelected = { dashboardViewModel.take(DashboardIntent.NavItemSelected(it)) }`
    - `onLogout` and `onProfileClick` stay as Composable-level callbacks (no change)

    **Koin DI update:** DashboardViewModel now takes `Sdk` as constructor parameter. `viewModelOf(::DashboardViewModel)` in AppModule.kt should auto-resolve Sdk from sdkModule. Verify this compiles. If Koin cannot auto-resolve (because DashboardViewModel previously had no-arg constructor), the `viewModelOf(::DashboardViewModel)` registration needs no code change -- Koin auto-resolves the single Sdk parameter.
