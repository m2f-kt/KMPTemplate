## Pre-Test Fixes

1. **AWS SDK version conflict** — Server failed to start with `NoClassDefFoundError: AwsSdkClientConfig$Builder`. Cause: Koog 0.6.2 brings `bedrockruntime:1.5.123` (uses `aws-core:1.5.123`) but S3 SDK was `1.4.45`. Fixed by upgrading `aws-s3` from `1.4.45` to `1.5.123`. Committed: `d66bc51`.

2. **Missing avatar_url column** — Login failed with "column users.avatar_url does not exist". The column wasn't created by Exposed auto-migration. Fixed manually with `ALTER TABLE users ADD COLUMN avatar_url VARCHAR(512)`.

3. **WASM deprecated asDynamic()** — Compilation warning about deprecated `asDynamic()`. Fixed by using typed `Uint8Array` with proper imports and `@OptIn(ExperimentalWasmJsInterop::class)`. Committed: `4405d63`.

4. **MinIO bucket policy missing public access** — Uploads succeeded (201 Created) but returned URLs were inaccessible (403 Access Denied). Root cause: the `uploads` bucket was missing the anonymous download policy. Fixed by adding `mc anonymous set download local/uploads` to docker-compose.yml minio-init service. Committed: `f556a73`.

5. **CORS missing localhost:8082** — WASM dev server started on port 8082 which wasn't in CORS allowlist. Fixed by adding `localhost:8082` to allowed origins. Committed: `19674f0`.

6. **Crop dialog image preview** — Crop dialog showed file size text instead of actual image. Created multiplatform `rememberDecodedImage` expect/actual function to decode ByteArray to ImageBitmap across all platforms. Committed: `93a3792`.

7. **Dashboard avatars not updating** — Dashboard sidebar and header showed hardcoded initials instead of using TerminalAvatar component. Also, navigating back from Profile didn't refresh avatar. Fixed by:
   - Adding `avatarUrl` to DashboardModel/Mutation/ViewModel
   - Using TerminalAvatar in DashboardSidebar and DashboardScreen
   - Using LifecycleEventObserver to trigger RefreshProfile on ON_RESUME
   - Removing redundant avatar from ProfileSidebar
   Commits: `b714c88`, `90e7f3f`, `d2d6683`.

8. **Multi-group UI hidden** — Creating additional groups when one exists caused UX issues (couldn't navigate between groups). Fixed by hiding "Create Group" button when a group already exists. Multi-group support deferred to future iteration.

9. **Missing invitations table** — Server returned 500 with exposed SQL error "relation invitations does not exist". Two fixes applied:
   - Manually created `invitations` table with all columns and indexes
   - Added StatusPages global exception handler to mask internal error details

10. **Invite link deep linking missing** — Two issues with invitation links:
    - APP_URL defaulted to port 8080 (server) instead of 8081 (web app)
    - WASM app had no deep link handling for `/invite/accept?token=...` URLs
    Fixed by:
    - Corrected APP_URL default to `http://localhost:8081`
    - Added `InviteLinkChecker` expect/actual (like `OAuthCallbackChecker`)
    - Added invite link detection in `App.kt` startup
    Commit: `e742812`

11. **WASM SPA routing for deep links** — Multiple issues fixed over several commits:
    - Added webpack `historyApiFallback` for SPA routing (`1f7db46`)
    - Used absolute paths for assets in index.html (`8168d40`)
    - Consolidated startup LaunchedEffects to prevent navigation race (`d42df53`)
    - Added webpack rewrites for Compose Resources (`834bb6c`)

12. **Compose Resources 404 on deep links** — Resources like fonts and strings failed to load when navigating to `/invite/accept?token=...` because browser requested them relative to current path (e.g., `/invite/composeResources/...`). Fixed by adding `<base href="/">` to index.html which tells the browser to resolve all relative URLs from root. Commit: `0520b94`

13. **MVI StateFlow race condition** — AcceptInvitation intent failed with "token is blank" because MviViewModel used lazy StateFlow initialization. Mutations sent before subscription were lost. Fixed by making `model` and `event` StateFlows eager (not `by lazy`) and using `SharingStarted.Eagerly` as default.

## Gaps

### Gap #1: Invitation Acceptance Flow for Unauthenticated Users

**Problem:** The current invitation flow assumes the user is already logged in when accepting. The `POST /api/invitations/accept` endpoint requires authentication (returns 401). This breaks the expected flow where:
- Admin invites user by email
- User (who may not have an account) receives email with invitation link
- User opens link and accepts

**Current (broken) flow:**
```
Admin invites → User gets email → User opens link → Accept button → 401 Unauthorized
```

**Required flow:**
```
Admin invites → User gets email → User opens link → 
  → If logged in + valid invitation → Link to group → Navigate to Admin Panel
  → If logged in + expired invitation → Show error + "Request new invitation" button
  → If not logged in + user exists → Navigate to Login (with invitation context) → After login, link to group
  → If not logged in + user doesn't exist → Navigate to Register (with invitation context) → After registration, link to group
  → If expired → Show error + "Request new invitation" option (no accept/register allowed)
```

**Required changes:**

1. **Server - New endpoint or modify existing:**
   - `GET /api/invitations/{token}` (public) - Already exists, returns invitation details
   - `POST /api/invitations/accept` - Needs to handle:
     - Authenticated user: Link to group (current behavior, but needs to work)
     - Unauthenticated user with existing account: Return indicator to login first
     - Unauthenticated user without account: Return indicator to register first
   - `POST /api/auth/register` - Accept optional `invitationToken` param, auto-link to group after registration
   - `POST /api/auth/login` - May need to accept invitation context for post-login linking

2. **Server - Security:**
   - Invitation token validation must check expiry
   - Registration with invitation token must validate token before allowing group link
   - Prevent group linking without valid invitation

3. **Client - InviteAcceptScreen:**
   - Check if user is logged in
   - If not logged in, show Login/Register options instead of Accept button
   - Pass invitation token through login/register flow
   - Handle expired invitation UI with "Request new invitation" option

4. **Client - LoginScreen/RegisterScreen:**
   - Accept optional invitation token parameter
   - After successful auth, if token present, call accept invitation API
   - Navigate to group admin panel on success

5. **Client - Navigation:**
   - `LoginRoute` and `RegisterRoute` need optional `invitationToken` param
   - Post-auth navigation needs to check for pending invitation

**Affected files:**
- `server/features/invitations/` - Accept endpoint logic
- `server/features/auth/` - Register endpoint to accept invitation token
- `app/auth/InviteAcceptScreen.kt` - Conditional UI for auth state
- `app/auth/InviteAcceptViewModel.kt` - Logic for auth state handling
- `app/auth/LoginScreen.kt` - Accept invitation param
- `app/auth/RegisterScreen.kt` - Accept invitation param
- `composeApp/navigation/Routes.kt` - Add invitation params to routes
- `composeApp/navigation/AppNavHost.kt` - Handle invitation flow routing

**Recommendation:** Create a new phase (18.2) to implement the complete invitation acceptance flow.
