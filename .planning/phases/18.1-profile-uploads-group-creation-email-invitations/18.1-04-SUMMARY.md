# Phase 18.1 Plan 04: Invitation System Backend Summary

Full-stack invitation system with token-based email invitations for group membership.

## What Was Built

### Database Layer
- **InvitationsTable**: Schema with id, token, groupId, email, invitedBy, role, expiresAt, acceptedAt, acceptedBy, createdAt columns
- **InvitationRepository**: CRUD operations with create(), findByToken(), markAccepted() using suspendTransaction pattern

### Service Layer
- **InvitationService**: Business logic with:
  - createInvitation(): Validates admin role, checks existing membership, 7-day expiry, sends email
  - getInvitation(): Public endpoint to view invitation details by token
  - acceptInvitation(): Validates token, checks expiry/acceptance, adds user to group

### API Routes
- `POST /api/groups/{groupId}/invitations/create` - Admin-protected, creates invitation
- `GET /api/invitations/{token}` - Public, returns invitation details for preview
- `POST /api/invitations/accept` - Auth-protected, accepts and joins group

### SDK Layer
- **InvitationApi**: Interface with createInvitation(), getInvitation(), acceptInvitation()
- **InvitationApiImpl**: HTTP implementation using type-safe resources
- **Sdk**: Updated facade to include InvitationApi delegation

### Testing Support
- **FakeInvitationApiBuilder**: DSL builder for test fakes following existing pattern
- **FakeSdkBuilder**: Updated to compose invitation API configuration

### Error Handling
- Added AppError.Invitation sealed class with NotFound, Expired, AlreadyAccepted variants
- Added InvitationErrors.kt with DomainError implementations
- Added gone() HTTP response helper for 410 Gone status
